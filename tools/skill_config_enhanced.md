Skill 配置页分为两个区域：左侧 **Front-matter 表单**编辑结构化元数据，右侧 **Section 列表编辑器**管理 SKILL.md 的 Markdown 章节体，支持章节新增、内容编辑与顺序调整。

**Front-matter 各字段设计逻辑：**

- **name / version** — Skill 的唯一标识与版本号。**【设计逻辑】** 版本化是 skill-router.md 精确检索的基础——路由引擎按 name 定位 Skill，按 version 决定是否触发更新；version 字段也是 deploy 快照的校验基准，确保项目运行时始终对应明确版本，支持回滚与可追溯。
- **category** — 按技术域分类（如 Serial / CAN / Power / Safety 等）。**【设计逻辑】** 分类标签是批量检索的索引键——路由引擎需要"所有 CAN 相关 Skill"时，category 可直接过滤，无需逐一扫描全量 Skill 内容，显著提升大规模资源池中的检索效率。
- **automotive_standards** — 关联的汽车行业规范（如 ISO 26262 / AUTOSAR / MISRA-C）。**【设计逻辑】** 合规要求前置声明使 Agent 调用 Skill 时自动感知约束，在代码生成阶段内建合规性校验，而非靠事后 Code Review 兜底——这是从"检测质量"升级为"内建质量"的关键设计。
- **use_cases** — 明确 Skill 适用的典型场景与边界条件。**【设计逻辑】** 场景边界比功能描述更重要——LLM 需要知道"什么时候不该用"才能避免误触发；显式 use_cases 约束可将 Skill 调用误触发率降至最低，也为日后 Skill 评测提供了可量化的精度基准。

**Section 章节体各项设计逻辑：**

- **Overview** — Skill 功能概述与适用前提。**【设计逻辑】** Overview 是路由引擎的"一句话摘要"——LLM 先读 Overview 快速判断场景是否命中，命中后再深入加载完整 Skill 内容，实现"按需加载"策略，避免无关 Skill 占用宝贵的上下文窗口。
- **Parameters** — 显式声明调用参数、数据类型与约束条件（Skill 的 API 合同）。**【设计逻辑】** 参数显式声明消除"猜测空间"——当输入输出有明确类型与约束时，LLM 生成的调用代码格式错误率可显著降低；参数段是提升 Skill 调用准确率贡献最高的结构化字段之一。
- **Code Examples** — 真实可运行的驱动代码模板（含具体芯片型号与寄存器配置）。**【设计逻辑】** 真实代码示例是提升生成准确率最直接的手段——LLM 遵循示例的准确率远高于从自然语言描述中推断；经实测，Code Examples 段是对最终代码输出质量贡献最大的单一章节。
- **Automotive Context** — 注入芯片型号、寄存器地址、时序约束等领域专属上下文。**【设计逻辑】** 领域专属上下文是消除 LLM 幻觉的关键——通用大模型不掌握具体芯片的寄存器映射，显式注入后可将"编造寄存器地址"的幻觉风险降至极低，这是汽车嵌入式场景中最不可或缺的上下文补偿机制。

Skill 遵循 **Karpathy Skill 规范**——高度结构化、参数显式声明、附有真实代码示例——确保 AI 调用时能精准理解每个 Skill 的用法边界，经实测可显著提升代码生成的准确率。
