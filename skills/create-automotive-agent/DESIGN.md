# create-automotive-agent — 核心设计思路

## 一、本质：把人类专家工作模式转化为可执行规范

这个 Skill 的根本目的是**角色抽象化**——将汽车软件工程师的职责边界、工作方式和判断逻辑，系统性地固化为 AI 可执行的 Agent 定义文件。它解决的核心问题是：如何让一个 AI Agent 能够像领域专家一样工作，而不只是回答问题。

---

## 二、三大设计支柱

### 1. 安全优先的强制约束链

ASIL 等级不是一个可选字段，而是贯穿整个 Agent 生命周期的**主线约束**：

```
信息收集（ASIL 等级）
  → system_prompt 模块 E（质量门禁：覆盖率 / peer review 要求）
    → escalation（ASIL-D 违规立即停止）
      → HUMAN CHECK（ASIL-D 安全唯一负责人触发人工介入）
```

每一层都不可绕过，形成闭合的安全保障环。这反映了汽车软件"安全不可降级"的行业本质。

### 2. 结构完整性即可靠性

硬性最低要求的背后是一个判断：**一个结构不完整的 Agent 定义，比没有 Agent 定义更危险**，因为它会给使用者虚假的安全感。

- system_prompt 必须有 A/B/C/D全部 4 个模块 → 缺少"上下文收集"模块的 Agent 会在信息不足时擅自推断
- Workflow 必须至少包含"主工作流 + 评审工作流" → 缺少评审流程的 Agent 无法支撑 ASIL-B 及以上的合规要求
- escalation 必须覆盖 3 条基础升级路径 → 缺少升级条件意味着 Agent 在遇到边界情况时无处"求援"

### 3. 人机协作边界的明确划定

3 个 HUMAN CHECK 条件定义了 AI 的**能力红线**：

| 红线 | 背后逻辑 |
|------|----------|
| ASIL-D 安全关键唯一负责人 | 安全关键决策不能由 AI 独担，必须有人类工程师背书 |
| 直接修改生产代码/配置的权限 | AI 执行生产变更的风险超出可接受范围 |
| "自动审批/跳过检查"等描述 | 这类描述本质上是在绕过行业安全机制 |

---

## 三、system_prompt 的五模块设计逻辑

5 个模块并非随意罗列，而是对应了一个专家的**完整认知闭环**：

```
A 核心身份   → 我是谁，我擅长什么（定位）
B 上下文收集 → 我需要知道什么才能开始工作（输入）
C 执行流程   → 我怎么工作（过程）
D 交付格式   → 我输出什么（结果）
E 质量门禁   → 什么是我的底线（约束）
```

缺少任何一个模块，Agent 的工作都会在某个环节失控。

---

## 四、追溯性是内建的，不是事后补充的

REQ → CODE → TEST 追溯矩阵被设计为**交付格式的强制项（模块 D）**，而不是可选输出。这个设计选择传递了一个重要信号：在汽车软件开发中，可追溯性不是审计时才需要的文档工作，而是工程工作本身的一部分。
